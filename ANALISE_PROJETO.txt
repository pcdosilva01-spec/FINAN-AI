PARTE 1 — VISÃO GERAL DO PROJETO
================================================================================

O que é o projeto?
------------------
FinLife AI é uma aplicação web de consultoria financeira pessoal. O usuário
preenche um wizard com seus dados (nome, idade, profissão, renda, gastos,
reserva, sonho e dificuldade), e depois conversa com uma IA chamada "Sofia"
que responde perguntas financeiras personalizadas com base nesses dados.

Tecnologias usadas:
-------------------
- HTML puro (index.html) — estrutura da interface
- CSS puro (style.css) — visual, dark mode, responsivo
- JavaScript puro (script.js) — toda a lógica do app
- Node.js (server.js e api/chat.js) — proxy para a API da NVIDIA NIM
- Vercel — hospedagem e serverless functions

Como o projeto funciona na prática:
------------------------------------
1. Usuário abre o site e vê a tela Splash
2. Clica em "Iniciar Diagnóstico" e vai para o Wizard (4 passos)
3. Passo 1: digita nome e idade
4. Passo 2: digita a profissão e clica em "Estimar Perfil" — o app busca
   dados locais de salário/gastos para aquela profissão
5. Passo 3: confirma ou edita renda, gastos e reserva atual
6. Passo 4: confirma ou edita meta financeira, sonho e dificuldade principal
7. Ao finalizar, vai para o app principal com sidebar + chat
8. A Sofia exibe um diagnóstico inicial local (sem API)
9. O usuário digita perguntas e a Sofia responde via API da NVIDIA NIM (streaming)

Onde os dados ficam salvos:
----------------------------
Tudo fica no localStorage do navegador. Não existe banco de dados, não existe
login, não existe servidor guardando nada. Se o usuário limpar o navegador,
perde tudo. A chave usada é "finlife_v6_state".

Arquitetura resumida:
---------------------
[Navegador]
  index.html → carrega style.css e script.js
  script.js → toda a lógica (UI, wizard, chat, IA, storage)

[Servidor / Vercel]
  api/chat.js → recebe POST do frontend, repassa para a API da NVIDIA NIM
  server.js → versão local do mesmo proxy (para rodar com Node.js localmente)

A separação faz sentido: o proxy existe para não expor a API key no frontend.

PARTE 2 — ARQUIVO: index.html
================================================================================

O que ele faz:
--------------
Define toda a estrutura visual do app em HTML estático. Não tem template
engine, não tem framework. É HTML puro com IDs que o JavaScript usa para
manipular o DOM.

Estrutura de telas (screens):
------------------------------
O app tem 3 "telas" que são divs com a classe .screen. Só uma fica visível
por vez (a que tem a classe .active). A troca é feita pelo JavaScript.

  #splash   — tela inicial com logo e botão "Iniciar Diagnóstico"
  #wizard   — formulário de cadastro em 4 passos
  #app      — interface principal com sidebar + chat

Elementos importantes e seus IDs:
-----------------------------------

SPLASH:
  btn-start → botão que inicia o wizard

WIZARD:
  wiz-bar / wiz-bar-fill → barra de progresso dos passos
  wiz-step-label → texto "Passo X de 4"
  wiz-prev / wiz-next → botões de navegação
  ws-1, ws-2, ws-3, ws-4 → os 4 passos do wizard (divs)
  f-name → input nome
  f-age → input idade
  f-prof → input profissão
  btn-estimate-profile → botão "Estimar Perfil com IA"
  f-income → input renda mensal
  f-cost → input gastos mensais
  f-reserve → input reserva atual
  f-goal → input meta de patrimônio
  f-dream → select sonho
  f-struggle → select dificuldade

APP PRINCIPAL:
  lbl-avatar → letra inicial do nome na sidebar
  lbl-name → nome do usuário na sidebar
  lbl-job → profissão + idade na sidebar
  lbl-income → renda formatada na sidebar
  lbl-cost → gastos formatados na sidebar
  lbl-surplus → sobra mensal na sidebar (verde ou vermelho)
  btn-reset → botão "Nova Análise"
  chat-feed → área onde as mensagens aparecem
  chat-form → formulário do input de chat
  chat-input → campo de texto do chat
  btn-send → botão de enviar mensagem

TOAST:
  toast → container da notificação flutuante
  toast-icon → emoji do tipo (✅ ⚠️ ❌ 💡)
  toast-title → título da notificação
  toast-body → texto da notificação
  toast-close → botão X para fechar
  toast-bar-fill → barra de progresso do tempo

O que está bom:
---------------
- Estrutura limpa e bem organizada
- IDs descritivos e consistentes
- Acessibilidade básica presente (aria-live, aria-label, role="log")
- Sem dependências externas (zero bibliotecas, zero CDN)

O que pode melhorar:
--------------------
- O campo f-reserve não tem "required", então o usuário pode deixar em branco
  e o app usa 0 — isso é intencional mas não está documentado no HTML
- O select f-dream tem uma option disabled como placeholder, mas o f-struggle
  também — se o usuário não selecionar nada e clicar em avançar, o required
  vai pegar, mas a mensagem de erro não é clara
- Não tem <meta og:> para compartilhamento em redes sociais
- Não tem favicon real (usa um data:image vazio)

O que pode ser cortado:
-----------------------
- Os dois glows ambientais (.glow-a e .glow-b) são puramente decorativos.
  Não prejudicam, mas adicionam dois elementos no DOM sem função.

  Busca o perfil local para uma profissão. Primeiro tenta match exato,

BLOCO 5 — TOAST NOTIFICATIONS
--------------------------------------------------------------------------------

Toast = { _t, _i, show, hide }

  Sistema de notificações flutuantes no canto inferior direito.
  
  show(type, title, body)
    type pode ser: "green", "amber", "red", "info"
    Mostra o toast com ícone, título e mensagem.
    Tem uma barra de progresso que esvazia em 4 segundos.
    Fecha automaticamente após 4 segundos.
  
  hide()
    Fecha o toast manualmente (botão X ou timeout).
  
  O que está bom: limpa timers anteriores antes de mostrar novo toast,
  então não acumula múltiplos toasts.
  
  Onde é usado:
  - Estimativa de perfil: "Analisando perfil..."
  - Perfil estimado com sucesso: "Perfil Estimado!"
  - Erro na estimativa: "Falha de conexão..."
  - Validação do wizard: "Por favor, digite sua ocupação primeiro."
  - Respostas da IA podem incluir [TOAST: tipo | título | mensagem] no texto

--------------------------------------------------------------------------------
BLOCO 6 — ANALYZER
--------------------------------------------------------------------------------

Analyzer = { compute(p, f) }

  Calcula métricas financeiras a partir do perfil (p) e finanças (f).
  
  O que calcula:
  - surplus: renda - gastos (sobra mensal)
  - savingRate: % da renda que sobra
  - idealReserve: gastos * 6 (reserva de emergência ideal = 6 meses de gastos)
  - reservePct: quanto da reserva ideal o usuário já tem (0 a 1)
  - grade: nota de A+ até D baseada na sobra e reserva
  - summary: texto descritivo da situação
  - conclusion: sugestão da Sofia
  
  Lógica das notas:
  - D: sobra negativa (gastando mais do que ganha)
  - C: sobra positiva mas menos de 10% da renda
  - B: economiza entre 10% e 20% da renda
  - A: economiza 20%+ e tem pelo menos 50% da reserva ideal
  - A+: economiza 35%+ e tem 100% da reserva ideal
  
  O que está bom: lógica simples, clara e financeiramente correta.
  A regra dos 6 meses de reserva é padrão no mercado financeiro.
  
  PROBLEMA: summary e conclusion são calculados mas nunca exibidos na
  interface atual. O sidebar não mostra mais o card de diagnóstico com
  grade e summary — esses dados existem mas ficam invisíveis.
  Ou você reativa o card de diagnóstico na sidebar, ou remove summary
  e conclusion do Analyzer para não calcular à toa.

--------------------------------------------------------------------------------
BLOCO 7 — NimAI (SERVIÇO DE IA)
--------------------------------------------------------------------------------

Este é o bloco mais complexo e também o mais problemático em termos de
código morto. Vou detalhar cada método:

getUserKey()
  Lê a chave de API customizada do localStorage.
  Funciona, mas como dito no Storage, não existe UI para o usuário
  definir essa chave. Código que não tem como ser ativado pelo usuário.

buildSystemPrompt(profile, finances, metrics)
  Monta o prompt do sistema enviado para a IA antes de cada conversa.
  Define o personagem Sofia, as regras de resposta e injeta os dados
  do usuário (renda, gastos, sobra, reserva, sonho, dificuldade).
  
  O que está bom: prompt bem estruturado, com regras claras para a IA
  não repetir formatos e responder de forma personalizada.
  
  PROBLEMA: as variáveis idealReserve e reservePct são calculadas mas
  nunca inseridas no prompt. Só surplus é usado. Pequena inconsistência.

normalizeFieldName(key)
  Mapeia nomes de campos em vários idiomas/formatos para nomes canônicos.
  Ex: "salario", "rendamensal", "monthlySalary" → todos viram "salary".
  
  PROBLEMA: esse método existe para processar respostas JSON da IA.
  Desde que a estimativa passou a ser local, ele nunca é chamado.
  Código morto no fluxo atual.

extractTextFromPayload(payload)
  Extrai texto de respostas da API em vários formatos possíveis.
  É uma função enorme (~60 linhas) que tenta extrair texto de qualquer
  estrutura de resposta imaginável.
  
  PROBLEMA: usada apenas no streamMessage para processar chunks de
  streaming. A maior parte da lógica (choices, output, data, response,
  result) nunca é atingida no fluxo de streaming atual, porque o
  streaming já vem em formato padrão OpenAI. Código defensivo demais.

extractMessageJSON, cleanJSONText, extractBracedJSON, parseJSONText,
findJSONText, parsePayloadObject, normalizeStruggle, normalizeDream,
validateProfilePayload, parseProfilePayload
  
  TODOS ESSES MÉTODOS são código morto no estado atual do projeto.
  Eles existiam para processar respostas JSON da IA quando a estimativa
  de perfil era feita via API. Desde que a estimativa passou a ser local,
  nenhum deles é chamado.
  
  Juntos somam aproximadamente 250 linhas de código que não fazem nada.
  Podem ser deletados sem afetar nada no app.

safeParseResponseBody(rawText)
  Tenta fazer JSON.parse de uma string, retorna a string se falhar.
  Também código morto no fluxo atual.

buildEstimationPrompt(profession, attempt, lastReason)
  Monta o prompt para pedir estimativa de salário à IA.
  Código morto — estimateProfile agora usa apenas dados locais.

extractSalaryFromText(text)
  Extrai um número de salário de um texto livre.
  Código morto — não é mais chamado.

localEstimateProfile(profession)
  Busca o perfil local e retorna income, cost, struggle, dream.
  ESTE é o método que realmente funciona e é usado.

estimateProfile(profession)
  Atualmente só chama localEstimateProfile. A versão anterior fazia
  uma chamada à API. A simplificação foi correta.

streamMessage(systemPrompt, messages, onChunk, onDone, onError)
  Faz a chamada real à API via streaming SSE (Server-Sent Events).
  É o único método do NimAI que realmente importa para o chat.
  
  Funciona assim:
  1. Monta o body com system prompt + histórico de mensagens
  2. Faz POST para /api/chat com stream: true
  3. Lê a resposta chunk por chunk
  4. Para cada linha "data: {...}", extrai o delta de texto
  5. Chama onChunk com cada pedaço de texto
  6. Quando termina, chama onDone
  7. Se der erro, chama onError
  
  O que está bom: implementação correta de SSE, com AbortController
  para timeout de 25 segundos.
  
  PROBLEMA: se a API retornar um erro HTTP (401, 429, 500), o código
  não trata isso — tenta ler o body como streaming e provavelmente
  não chama onError, deixando o chat travado. Deveria checar
  response.ok antes de tentar ler o stream.


--------------------------------------------------------------------------------
BLOCO 8 — UI CONTROLLER
--------------------------------------------------------------------------------

UI = { state, chatHistory, isSending, isEstimating, ... }

  Objeto central que controla toda a interface. Tem estado interno e
  métodos para cada ação do usuário.

state
  Objeto com { profile, finances }. É o dado central do app.
  profile: { name, age, profession, income, cost, goal, dream, struggle }
  finances: { reserve, netWorth }
  
  PROBLEMA: netWorth é sempre igual a reserve. Existe no objeto mas nunca
  é calculado de forma diferente. Se no futuro quiser adicionar outros
  ativos (investimentos, etc), faz sentido. Por ora é redundante.

chatHistory
  Array de { role, content } — histórico de mensagens enviadas para a IA.
  Salva as últimas 20 mensagens no localStorage.
  Enviado para a API nas últimas 10 mensagens (contexto da conversa).

init()
  Inicializa o app. Verifica se tem estado salvo no localStorage.
  Se sim, vai direto para o app. Se não, mostra a splash.
  Correto e bem feito.

showScreen(id) / showSplash() / showWizard() / showApp()
  Controla qual tela está visível. Remove "active" de todas e adiciona
  na tela alvo. Simples e funciona bem.

syncSidebar()
  Atualiza os dados exibidos na sidebar (nome, profissão, renda, gastos,
  sobra). Chamado ao entrar no app e após qualquer mudança de estado.
  Correto.

setWizardStep(value)
  Muda o passo atual do wizard. Quando vai para o passo 4, recalcula
  automaticamente a meta sugerida com base nos valores do passo 3.
  Faz sentido e resolve o bug que existia antes.

renderChatHistory()
  Renderiza o histórico salvo quando o usuário volta ao app.
  Correto.

appendAIIntro()
  Exibe a mensagem inicial da Sofia ao entrar no app.
  Calcula sobra, prazo para meta, e dicas baseadas na dificuldade.
  Tudo local, sem API. Bem feito.
  
  PROBLEMA menor: se o usuário voltar ao app (já tem histórico salvo),
  essa função não é chamada — o histórico é renderizado no lugar.
  Mas se o histórico estiver vazio por algum motivo, ela é chamada.
  Comportamento correto.

saveCurrentState()
  Salva estado + histórico no localStorage após cada mensagem.
  Correto.

appendMessage(role, sender, html)
  Cria um balão de mensagem no chat feed.
  
  PROBLEMA de segurança: para mensagens do usuário (role === "user"),
  o código faz escapeHTML(html) — correto. Mas o parâmetro já se chama
  "html" e vem de userText, que já é texto puro. Funciona, mas o nome
  do parâmetro é confuso — deveria se chamar "content" para deixar
  claro que para user é texto, para ai é HTML.

sendMessage(userText)
  Método principal do chat. Fluxo:
  1. Desabilita input e botão
  2. Adiciona mensagem do usuário no feed
  3. Cria balão vazio da Sofia com "Analisando seus dados..."
  4. Inicia timer que troca o texto do placeholder a cada 1,2s
  5. Chama NimAI.streamMessage com o prompt e histórico
  6. A cada chunk recebido, atualiza o balão com o texto parcial
  7. Ao terminar, remove cursor, verifica se tem [TOAST:...] no texto
  8. Salva resposta no histórico e reabilita o input
  
  O que está bom: experiência de streaming bem implementada, scroll
  inteligente (só rola se o usuário já estava no final).
  
  PROBLEMA: se a API retornar erro HTTP (não AbortError), o onError
  é chamado e mostra a mensagem de erro no balão. Mas isSending fica
  false e os botões são reabilitados — isso está correto. Porém o
  histórico não salva a mensagem de erro, então o usuário pode tentar
  de novo sem problemas. Comportamento aceitável.

filterToastTag(text)
  Procura o padrão [TOAST: tipo | título | mensagem] no texto da IA.
  Se encontrar, remove do texto e retorna separado para exibir como toast.
  
  Ideia interessante mas problemática na prática: a IA raramente vai
  gerar esse formato espontaneamente, e o prompt atual não instrui a IA
  a usar esse formato. Na prática, esse filtro nunca dispara.
  Pode ser removido ou o prompt precisa instruir a IA a usá-lo.

renderMarkdown(raw)
  Converte Markdown básico para HTML para exibir as respostas da Sofia.
  Suporta: h1/h2/h3, blockquote, listas, negrito, itálico, código inline,
  blocos de código, ✔/❌/✅/⚠️ com classes coloridas.
  
  O que está bom: funciona bem para o caso de uso.
  
  PROBLEMA 1: tem regex hardcoded para títulos específicos como
  "## 📊 Seu diagnóstico" e "## ✅ O que você fez certo". Esses títulos
  eram do prompt antigo que foi removido. Essas linhas são código morto.
  
  PROBLEMA 2: a regex de listas agrupa todos os <li> em um único <ul>,
  mas se houver duas listas separadas por texto no meio, elas vão ser
  fundidas em uma só. Bug raro mas possível.
  
  PROBLEMA 3: escapa HTML antes de processar Markdown, o que é correto
  para segurança. Mas isso significa que emojis e caracteres especiais
  no texto da IA passam sem problema.

bindEvents()
  Registra todos os event listeners. Bem organizado.
  
  PROBLEMA: o confirm() nativo do browser para "Nova Análise" é feio
  e inconsistente entre browsers. Poderia ser um modal customizado.

PARTE 5 — ARQUIVO: api/chat.js (PROXY SERVERLESS)
================================================================================

O que ele faz:
--------------
É uma Vercel Serverless Function. Recebe POST do frontend, repassa para
a API da NVIDIA NIM e devolve a resposta (streaming ou não).

Por que existe:
---------------
Para não expor a API key no frontend. A key fica em variável de ambiente
no Vercel (NVIDIA_NIM_API_KEY) e nunca chega ao browser.

Fluxo:
------
1. Recebe POST com body JSON
2. Lê a API key do header X-Override-Key ou da env NVIDIA_NIM_API_KEY
3. Injeta o model ID (NVIDIA_NIM_MODEL_ID ou "openai/gpt-4o-mini")
4. Faz proxy da requisição para integrate.api.nvidia.com
5. Faz pipe da resposta de volta para o cliente

O que está bom:
---------------
- CORS configurado corretamente
- Trata OPTIONS (preflight) corretamente
- Pipe direto da resposta (não bufferiza tudo na memória)
- Trata erro de conexão com o servidor da NVIDIA

O que pode melhorar:
--------------------
- Não valida o body recebido além do JSON.parse básico. Um body malformado
  ou com campos inesperados vai direto para a NVIDIA sem sanitização.

- Não tem rate limiting. Qualquer pessoa que descobrir a URL do proxy pode
  fazer requisições ilimitadas usando sua API key.

- Se NVIDIA_NIM_API_KEY não estiver configurada, retorna 401. Mas a mensagem
  de erro "Nenhuma API key configurada." vai aparecer no chat como erro de
  conexão, sem deixar claro para o usuário o que fazer.

- O model ID padrão é "openai/gpt-4o-mini" — mas o endpoint é da NVIDIA NIM,
  não da OpenAI. Isso funciona porque a NVIDIA NIM tem compatibilidade com
  a API da OpenAI, mas o nome do modelo pode confundir quem for manter o código.
